# 项目启动与开发准备说明

## 1. 编写目的
本文档用于明确竞赛管理系统进入正式开发阶段前的组织方式、开发边界、协作规则、环境准备、交付基线与风险预案，为需求、设计、开发、测试、部署与验收提供统一起点。

## 2. 项目背景
现有竞赛组织工作普遍存在报名信息分散、材料提交方式不统一、资格调整依赖人工记录、赛前提醒与赛后整理依赖经验记忆等问题。为降低重复性人工操作，提高流程可追溯性与数据一致性，项目拟建设平台内闭环的竞赛管理系统，将报名、审核、状态流转、通知、统计和归档统一纳入平台管理。

## 3. 项目目标
### 3.1 建设目标
- 建立统一的赛事与学生信息管理入口
- 建立从报名到赛后归档的完整流程闭环
- 降低重复录入、人工核对和手工整理成本
- 建立可审计的操作日志、状态记录与权限控制
- 形成适用于 Windows 运行环境的简洁部署方案

### 3.2 范围界定
本项目纳入正式开发范围的内容包括：
- 赛事信息管理
- 学生基础信息管理
- 在线报名与批量导入导出
- 材料提交、审核与补正
- 资格确认、退赛、换人、补录等状态流转
- 平台内消息与通知中心
- 赛后成绩、证书与归档管理
- 统计报表与资料导出

## 4. 建设原则
### 4.1 业务原则
- 平台作为业务处理唯一主链路
- 报名、审核、状态变更与归档均在平台内完成
- 外部渠道仅承担补充触达，不承载业务处理

### 4.2 技术原则
- 前后端分离
- 前端基于 Ruoyi-Vue 原生管理端结构复用
- 后端固定采用 Flask 实现
- 兼顾 macOS 开发环境与 Windows 运行环境
- 优先使用成熟、资料较充分、维护成本可控的技术组件

### 4.3 交付原则
- 以完整业务闭环为目标组织开发
- 以可维护、可测试、可部署、可验收为交付基线
- 不以最小化原型替代正式系统范围

## 5. 组织与角色建议
| 角色 | 主要职责 |
|---|---|
| 项目负责人 | 统筹需求、计划、资源与对外沟通 |
| 产品/需求负责人 | 梳理业务流程、输出需求规格与原型 |
| 前端开发 | 基于 Ruoyi-Vue 完成页面、组件、权限与交互开发 |
| 后端开发 | 基于 Flask 完成接口、业务规则、权限、日志与任务能力 |
| 数据库/接口负责人 | 维护表结构、索引、接口契约与数据字典 |
| 测试负责人 | 用例设计、功能验证、回归测试、验收支持 |
| 运维/部署支持 | Windows 环境部署、容器编排、日志与备份检查 |

## 6. 开发环境准备
### 6.1 基础环境
- 开发终端：macOS
- 代码托管：Git
- 协作方式：分支管理 + 合并请求 + 代码评审
- 接口联调：本地开发环境 + 测试环境
- 数据库版本：MySQL 8.0

### 6.2 前端环境
- Node.js LTS 版本
- 包管理工具：npm 或 pnpm
- 开发基础：Vue3、Element Plus、Ruoyi-Vue 前端基础结构

### 6.3 后端环境
- Python 3.10 及以上版本
- Flask
- 数据访问建议：SQLAlchemy
- 数据校验建议：Pydantic 或 Marshmallow
- 定时任务建议：APScheduler 或 Celery（视部署方式确定）
- 文档生成建议：OpenAPI/Swagger 兼容方案

## 7. 运行与部署准备
### 7.1 主要运行环境
- 操作系统：Windows
- 推荐方式：Docker Desktop + Docker Compose
- 替代方式：Python 虚拟环境 + MySQL 本地服务 + 前端静态资源部署

### 7.2 部署要求
- 单机部署流程明确
- 环境变量配置集中管理
- 初始化脚本可重复执行
- 日志目录、上传目录、备份目录可配置
- 数据库备份与恢复流程具备可操作性

## 8. 协作与版本管理
### 8.1 分支建议
- `main`：稳定发布分支
- `develop`：日常集成分支
- `feature/*`：功能开发分支
- `fix/*`：缺陷修复分支

### 8.2 提交与评审要求
- 提交信息应包含模块与变更摘要
- 涉及数据库或接口变更时需同步更新文档
- 合并前需完成基本自测与同伴评审

## 9. 风险与应对
| 风险项 | 说明 | 应对措施 |
|---|---|---|
| 需求边界蔓延 | 临时新增需求影响排期 | 建立需求变更登记与评审机制 |
| 前后端契约不一致 | 页面与接口字段口径不统一 | 优先维护接口文档与数据字典 |
| Windows 运行适配问题 | 开发环境与运行环境差异 | 提前完成 Windows 部署验证 |
| 数据导入质量不足 | 历史数据不规范 | 增加导入模板校验与错误回显 |
| 流程覆盖不完整 | 仅覆盖报名而未覆盖后续环节 | 以完整闭环清单作为范围基线 |

## 10. 启动阶段输出物
正式开发开始前建议完成以下输出：
1. 项目启动与开发准备说明
2. 需求规格说明书
3. 总体技术方案与系统架构设计说明书
4. 数据库与接口设计说明书
5. 开发计划、测试与验收方案
6. 页面原型或页面清单
7. 初始数据字典
8. 部署准备清单

## 11. 结论
项目已从“工具性辅助”方向调整为“平台内闭环管理”方向。正式开发阶段应以完整流程、统一数据口径、明确责任边界和稳定部署能力为基本前提，确保后续编码、联调、测试与验收具有一致依据。
