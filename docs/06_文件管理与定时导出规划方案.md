# 文件管理与定时导出规划方案

## 1. 编写目的
本文档用于规划竞赛管理系统下一阶段的“文件管理中心 + 定时导出 + 外部云盘投递”能力，重点解决平台端材料分散、导出动作依赖人工、归档结构不统一、外部存储交付能力缺失的问题。

## 2. 当前系统基础与现状
### 2.1 已具备的基础能力
- 平台已具备附件上传、证书归档、统计导出、备份恢复等基础文件能力
- 统计导出中心已支持异步任务、进度轮询、失败重试与重启恢复
- 系统参数页已具备存储与备份运行态展示入口

### 2.2 当前不足
- 文件能力仍以业务页面内嵌附件为主，缺少统一的文件管理中心
- 导出中心目前偏“单次任务”，缺少按规则自动生成、自动归类、自动投递能力
- 存储驱动仍只有本地 `local`，没有对象存储或第三方云盘适配层
- 现有导出更偏“报表文件”，不适合做“多类材料的分级归档包”
- 暂无“每日定时导出”调度中心，也没有“导出后投递到百度网盘”的渠道管理能力

## 3. 建设目标
本次规划建议新增一个平台级文件管理子系统，目标如下：

1. 在平台端形成统一的文件管理页面，按类别、赛事、学院、日期、任务批次查看和检索材料。
2. 支持按预设规则每日自动导出各类材料，不依赖人工逐个点击。
3. 导出结果在页面内可见，且自动按类别和目录规则分级整理。
4. 支持配置导出模式，例如“按赛事归档”“按学院归档”“按日期快照归档”“按业务类别归档”。
5. 支持将导出产物自动投递到百度网盘指定目录。
6. 对导出、投递、失败、重试、覆盖、历史版本具备完整审计记录。

## 4. 建设范围
### 4.1 平台文件管理中心
- 统一展示平台内各类业务附件与导出产物
- 支持目录树、分类筛选、关键字搜索、批次查看、下载和预览
- 支持查看文件来源，如报名材料、规则附件、证书、统计报表、归档包、系统备份

### 4.2 定时导出中心
- 支持配置每日定时任务
- 支持全量导出、增量导出、按条件导出
- 支持任务执行日志、失败重试、下一次执行时间展示

### 4.3 导出规则与目录模板
- 支持定义导出内容、筛选条件、文件命名规则、目录结构规则
- 支持一套规则生成多个渠道产物，如本地归档包和百度网盘同步目录

### 4.4 外部投递渠道
- 第一阶段支持本地归档目录
- 第二阶段支持百度网盘
- 后续可扩展到对象存储、NAS、SFTP 等渠道

## 5. 目标场景
### 场景一：每日自动归档报名材料
- 每天凌晨自动导出当天新增或变更的报名材料
- 平台页面中按“报名材料 / 赛事 / 日期”自动归档展示
- 同步推送到百度网盘指定目录

### 场景二：每日导出赛后证书与成绩归档
- 每天固定时间汇总新增成绩记录、证书文件和统计报表
- 生成标准目录结构，供学院或竞赛负责人查阅

### 场景三：按学院生成分发包
- 每天根据学院维度自动生成归档目录
- 学院负责人只查看自己有权限的文件包

### 场景四：运营管理员查看历史导出批次
- 能看到每次自动导出的批次、范围、文件数量、失败原因、百度网盘投递结果

## 6. 功能设计
### 6.1 文件管理中心
建议新增一级菜单：`文件中心`

建议包含 3 个页面：

1. `文件总览`
- 目录树浏览
- 文件列表
- 分类筛选
- 来源筛选
- 预览/下载

2. `导出规则`
- 新增/编辑规则
- 配置导出范围
- 配置目录模板
- 配置命名模板
- 配置投递渠道

3. `导出任务`
- 手工执行
- 定时执行记录
- 当前运行状态
- 失败重试
- 投递日志

### 6.2 文件分类模型
建议先统一定义以下一级分类：
- 报名材料
- 赛事规则
- 成绩附件
- 证书文件
- 统计报表
- 赛后归档
- 系统备份
- 其他平台文件

二级分类由业务规则生成，例如：
- 报名材料 / 省级程序设计竞赛 / 2026-03-15
- 证书文件 / 数据分析挑战赛 / 商学院
- 赛后归档 / 按学院 / 计算机学院

### 6.3 导出规则配置项
每条规则建议包含：
- 规则名称
- 启用状态
- 导出类型
- 导出频率
- 执行时间
- 导出范围
- 增量或全量模式
- 目录模板
- 文件命名模板
- 是否打包为 ZIP
- 是否生成清单文件
- 投递渠道
- 投递目标目录
- 失败处理策略

### 6.4 导出模式建议
建议首批支持以下模式：

1. 按业务类别归档
- 例如：`报名材料/赛事/日期/文件`

2. 按赛事归档
- 例如：`赛事名称/报名材料`
- 例如：`赛事名称/证书文件`
- 例如：`赛事名称/统计报表`

3. 按学院归档
- 例如：`学院/赛事/材料类型/文件`

4. 按日期快照归档
- 例如：`2026/03/15/报名材料`
- 适合每日固定快照导出

5. 按任务批次归档
- 例如：`批次号/清单.json`
- 适合审计和问题追溯

### 6.5 页面呈现方式
页面建议采用“左树右表 + 顶部任务卡片”的结构：

- 左侧：分类目录树
- 右侧上方：今日导出概览、最近一次成功、待重试任务、百度网盘同步状态
- 右侧中部：文件列表或批次列表
- 右侧下方：任务详情、执行日志、投递记录

### 6.6 审计与权限
- 管理员可看全部
- 教师仅看自己负责赛事范围内的导出数据
- 审核员默认只读，不可改规则
- 文件列表与导出结果继续叠加现有赛事权限与学院数据权限
- 所有规则变更、手工执行、失败重试、云盘投递都写入操作日志

## 7. 百度网盘对接规划
### 7.1 对接目标
不是把百度网盘当主存储，而是把它作为“导出投递渠道”。

也就是说：
- 平台主数据和附件主存储仍在平台侧
- 导出任务完成后，把导出产物同步到百度网盘指定目录
- 平台保留导出记录、投递状态和文件清单

### 7.2 建议对接方式
建议采用“渠道适配器”设计，不把百度网盘逻辑写死到导出任务里。

建议抽象统一接口：
- `prepare_target()`
- `ensure_folder()`
- `upload_file()`
- `upload_manifest()`
- `query_result()`
- `retry_delivery()`

第一批只实现：
- 百度网盘目录检查/创建
- 文件上传
- 执行结果回写

第二批再考虑：
- 覆盖策略
- 断点续传
- 大文件分片
- 已存在文件比对
- 历史版本保留

### 7.3 配置项建议
每个百度网盘渠道建议配置：
- 渠道名称
- 启用状态
- 认证方式
- 应用标识
- 授权信息或令牌信息
- 目标根目录
- 是否按规则自动创建目录
- 上传冲突策略
- 上传失败重试次数

说明：具体认证与上传字段应以百度网盘开放平台最新官方接口为准，实施阶段再按官方文档确定请求细节。

## 8. 技术实现建议
### 8.1 模块划分
建议新增以下后端模块：

1. `file_center`
- 文件元数据管理
- 分类树生成
- 文件检索与列表

2. `export_policy`
- 导出规则管理
- 目录模板管理
- 命名模板管理

3. `export_scheduler`
- 每日调度
- 批次生成
- 恢复执行

4. `export_packager`
- 文件收集
- 目录结构构建
- ZIP 打包
- 清单文件生成

5. `delivery_channel`
- 本地目录投递
- 百度网盘投递

### 8.2 存储层建议
当前系统的 `AttachmentInfo` 更偏“单个业务附件”，建议新增“文件中心元数据层”，不要直接把所有页面都改成依赖物理路径。

建议引入以下核心表：
- `file_asset`
- `file_category`
- `file_export_policy`
- `file_export_batch`
- `file_export_batch_item`
- `file_delivery_channel`
- `file_delivery_record`

### 8.3 任务执行建议
当前导出中心已经支持异步任务与重启恢复，但仍是进程内执行器。

本需求建议升级为真正的任务调度模型：
- 规则任务
- 执行批次
- 投递任务
- 恢复扫描
- 幂等控制

单机版可先继续沿用数据库 + 应用内调度；
如果后续需要多实例部署，建议切到独立 worker 或基于 Redis 的队列。

### 8.4 清单文件建议
每次导出建议生成一个 `manifest.json` 或 `manifest.xlsx`，记录：
- 批次号
- 规则名称
- 执行时间
- 文件总数
- 文件来源
- 文件大小
- 原始业务对象编号
- 目标路径
- 投递状态

这样平台页和外部投递都能共用同一份清单。

## 10. 实施阶段建议
### 阶段一：平台文件中心基线
目标：
- 先把现有报名材料、证书、规则附件、导出报表统一纳入文件中心展示

交付：
- 文件总览页
- 文件分类树
- 文件资产表
- 文件检索接口

### 阶段二：导出规则与每日任务
目标：
- 支持每日自动生成归档批次

交付：
- 导出规则管理页
- 导出批次页
- 每日调度执行
- 目录模板和命名模板

### 阶段三：百度网盘投递
目标：
- 支持将导出结果自动同步到百度网盘

交付：
- 渠道配置页
- 百度网盘投递适配器
- 投递日志与失败重试

### 阶段四：增强能力
目标：
- 提升可运维性和大批量处理能力

交付：
- 取消任务
- 清理策略
- 归档版本保留
- 大文件优化
- 多实例 worker 方案

## 11. 验收建议
建议按以下口径验收：

1. 平台页面能统一看到文件，并按类别分级展示。
2. 每日定时任务能自动生成导出批次。
3. 导出目录结构符合配置模板。
4. 导出结果能在平台内下载、查看清单、查看失败原因。
5. 百度网盘投递成功后，平台可看到目标路径和投递状态。
6. 失败任务可重试，重启后不中断。
7. 权限口径与现有赛事/学院数据权限保持一致。

## 12. 风险与注意事项
### 12.1 百度网盘接口风险
- 接口能力、认证方式、频控与文件大小限制需以官方最新文档为准
- 规划阶段不要把百度网盘当唯一存储

### 12.2 目录规则复杂度风险
- 如果目录模板过于灵活，会明显增加实现复杂度
- 第一阶段建议只支持有限模板变量，不做任意表达式引擎

### 12.3 数据口径风险
- 文件中心不能直接绕开现有权限体系
- 必须复用赛事权限和学院权限，否则会引入越权查看问题

### 12.4 大批量导出性能风险
- 大量文件打包和上传会占用 IO 和 CPU
- 需要提前设计批次拆分、并发上限和失败恢复机制

## 13. 建议结论
该需求适合按“文件中心基线 -> 定时导出 -> 百度网盘投递 -> 增强运维能力”四步推进。

不建议一开始就把百度网盘做成主存储。
建议先把平台内的文件资产、导出规则、批次模型和目录组织规则做稳，再接百度网盘渠道。

这样做的好处是：
- 平台内先形成统一文件闭环
- 外部渠道只是投递适配，不会反向绑死业务
- 后续若改接对象存储、NAS、SFTP，也不用重做业务层
