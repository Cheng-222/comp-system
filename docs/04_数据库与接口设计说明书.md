# 数据库与接口设计说明书

## 1. 编写目的
本文档用于说明竞赛管理系统核心数据结构、表关系、字段建议、接口规范、权限控制与日志要求，为数据库实施、后端开发、前端联调和测试设计提供依据。

## 2. 数据设计原则
- 以业务对象为中心进行实体建模
- 审计字段统一
- 状态字段统一枚举或字典管理
- 关键关系具备外键或逻辑约束
- 导入、审核、流转等高频操作优先考虑索引设计
- 兼顾规范化与查询效率

## 4. 审计字段规范
建议所有业务表统一包含以下字段：
- `created_at`
- `updated_at`
- `created_by`（如适用）
- `updated_by`（如适用）
- `deleted_flag`（逻辑删除时适用）

## 6. 接口设计规范
### 6.1 基本规范
- 风格：RESTful
- 数据格式：JSON
- 编码：UTF-8
- 版本建议：`/api/v1`
- 分页参数：`pageNum`、`pageSize`
- 通用查询参数：关键字、状态、时间区间、赛事ID、学生ID

### 6.2 返回结构建议
```json
{
  "code": 200,
  "message": "success",
  "data": {},
  "traceId": "optional"
}
```

### 6.3 错误处理
- 参数错误
- 权限不足
- 业务状态不允许
- 资源不存在
- 文件处理失败
- 导入导出失败

## 7. 典型接口清单
### 7.1 赛事管理
- `GET /api/v1/contests`
- `POST /api/v1/contests`
- `GET /api/v1/contests/{id}`
- `PUT /api/v1/contests/{id}`
- `POST /api/v1/contests/{id}/publish`

### 7.2 学生管理
- `GET /api/v1/students`
- `POST /api/v1/students/import`
- `POST /api/v1/students`
- `PUT /api/v1/students/{id}`

### 7.3 报名管理
- `GET /api/v1/registrations`
- `POST /api/v1/registrations`
- `GET /api/v1/registrations/{id}`
- `POST /api/v1/registrations/import`
- `GET /api/v1/registrations/export`

### 7.4 材料审核
- `POST /api/v1/registrations/{id}/materials`
- `POST /api/v1/registrations/{id}/review`
- `POST /api/v1/registrations/{id}/correction`

### 7.5 状态流转
- `POST /api/v1/registrations/{id}/withdraw`
- `POST /api/v1/registrations/{id}/replace`

### 7.6 通知中心

- `POST /api/v1/messages`
- `POST /api/v1/messages/{id}/send`

### 7.7 赛后管理
- `POST /api/v1/results/import`
- `GET /api/v1/results`
- `POST /api/v1/certificates/upload`
- `GET /api/v1/statistics/awards`

## 8. 权限控制建议
- 页面权限：菜单与路由显隐
- 按钮权限：新增、编辑、审核、导出、状态调整等
- 数据权限：按角色、赛事范围、学院范围控制可见数据
- 接口权限：前后端双重校验

## 9. 索引建议
- `student_info.student_no`
- `contest_registration.contest_id + student_id`
- `contest_registration.final_status`
- `registration_material.registration_id`
- `registration_flow_log.registration_id`
- `contest_result.contest_id + student_id`
- `notice_message.send_status`

## 10. 结论
数据库与接口设计应围绕完整业务闭环建立统一的数据主线，确保报名、审核、状态流转、通知、赛后归档与统计输出之间能够形成稳定的数据关联与审计链路。
