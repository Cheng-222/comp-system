# 总体技术方案与系统架构设计说明书

## 1. 编写目的
本文档用于说明竞赛管理系统的技术选型、总体架构、模块划分、部署策略、技术约束与扩展边界，为系统实现提供统一的工程设计依据。

## 2. 技术选型
| 层级 | 选型 | 说明 |
|---|---|---|
| 前端 | Vue3 + Element Plus + Ruoyi-Vue 原生管理端结构 | 复用成熟管理端布局、权限、路由、表格与表单组织方式 |
| 后端 | Flask | 适配项目技术要求，轻量、灵活，便于构建 RESTful API |
| 数据库 | MySQL 8.0 | 适用于结构化业务数据、检索、统计与状态管理 |
| 认证与权限 | 基于 Token/Session 的认证方案 | 结合角色、菜单、按钮级权限控制 |
| 文件存储 | 本地文件系统或对象存储适配层 | 根据部署环境选择，保持接口一致 |
| 缓存/队列 | 视规模引入 Redis | 用于缓存、通知任务、限流或延迟任务 |
| 部署 | Docker Compose 优先 | 简化 Windows 部署适配与环境一致性 |
| Web 服务 | Nginx（前端静态资源/代理）+ Flask 应用服务 | 保持前后端分离与统一入口 |

## 3. 选型说明
### 3.1 前端方案
前端不采用完全自定义后台框架，基于 Ruoyi-Vue 原生管理端结构进行二次开发，复用其菜单、布局、权限、标签页、表格工具栏等成熟组织方式。在此基础上，接口层统一适配 Flask 后端提供的 API。

### 3.2 后端方案
后端采用 Flask 构建业务服务，建议按蓝图分模块组织，保证竞赛、报名、审核、通知、统计等模块之间边界清晰。对外统一提供 RESTful API，内部保持服务层、数据访问层与任务处理层分离。

### 3.3 Windows 运行环境适配
考虑主要运行环境为 Windows，推荐采用 Docker Compose 作为部署主方案，以减少 Python 依赖、数据库版本与系统环境差异带来的适配成本。保留本地安装方案作为补充，满足单机调试与教学环境使用。

## 4. 总体架构
```text
用户层
  ├─ 学生用户
  ├─ 竞赛管理员/教师
  └─ 系统管理员

表现层
  └─ Ruoyi-Vue 管理端 / 学生侧页面

服务层
  └─ Flask API 服务
      ├─ 认证与权限模块
      ├─ 学生管理模块
      ├─ 赛事管理模块
      ├─ 报名管理模块
      ├─ 材料审核模块
      ├─ 资格流转模块
      ├─ 通知中心模块
      ├─ 赛后归档模块
      └─ 统计报表模块

数据层
  ├─ MySQL
  ├─ 文件存储目录 / 对象存储
  └─ 日志与备份目录
```

## 5. 分层设计
### 5.1 表现层
- 承担页面展示、交互组织、数据校验提示与权限显隐
- 不承载复杂业务规则
- 与后端通过统一 API 模块通信

### 5.2 接口层
- 负责请求接入、参数校验、鉴权与响应格式统一
- 控制错误码、分页格式、导入导出任务结果格式

### 5.3 业务服务层
- 负责报名规则、审核规则、状态流转、通知生成、统计计算等业务逻辑
- 保持与页面实现解耦

### 5.4 数据访问层
- 负责实体映射、查询封装、事务控制与数据库操作
- 统一管理索引策略、数据软删除与审计字段

## 7. 关键设计要点
### 7.1 平台内闭环
系统以平台为唯一业务主链路。通知能力以平台内消息、公告、待办为主。若存在外部触达需求，应通过可替换的消息适配层处理，不进入主业务状态采集与流转逻辑。

### 7.2 统一状态模型
报名、审核、资格、赛后归档等业务环节应采用统一状态机设计，避免同一记录在不同页面出现状态口径不一致。

### 7.3 统一日志模型
关键动作包括导入、审核、退赛、换人、补录、导出、证书归档等，均应记录操作人、时间、对象、动作、前后状态、结果。

### 7.4 界面一致性
基于 Ruoyi-Vue 的原生布局与表格体系，统一页面栅格、留白、组件尺寸、按钮位置与详情页结构，保证管理端界面整洁、稳定、易于维护。

## 8. 部署架构建议
### 8.1 推荐部署
- 前端：构建为静态资源，由 Nginx 承载
- 后端：Flask 服务容器
- 数据库：MySQL 容器
- 文件挂载：Windows 本地目录映射
- 日志：独立挂载目录

### 8.2 备选部署
- 前端构建后部署至 IIS 或 Nginx
- Flask 使用 Waitress 运行
- MySQL 采用本地安装方式
- 适用于小规模、单机环境

## 9. 扩展边界
后续如需增加邮件、短信、企业微信、学校统一身份认证等能力，应通过接口适配层扩展，不改变核心业务结构。

## 10. 不建议纳入主链路的能力
- 依赖桌面环境的消息监听
- 基于聊天记录的业务状态采集
- 难以验证责任边界的自动代操作能力

## 11. 结论
本方案采用“Ruoyi-Vue 前端结构复用 + Flask 后端服务 + MySQL 数据存储 + Docker Compose 优先部署”的技术路线，能够兼顾开发效率、Windows 运行环境适配与完整业务闭环目标。
